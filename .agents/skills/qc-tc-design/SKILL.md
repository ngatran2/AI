# Test Case Design Skill

## Purpose

Read the latest version of the UC Readiness Report and Test Scenarios (if available), To systematically design test cases for any given feature. This skill operates in **two distinct modes**:

### Mode: `high-level` (HL)
- **Goal:** Generate 5–15 representative test cases for Smoke/Sanity testing.
- **Focus:** Happy Path per sub-function + 1 Critical Error per sub-function + 1 RBAC check.
  - **Draft/Low Detail Rule:** If input is a **Draft** or score is **70-89**, strictly focus **ONLY on Main Flow / Happy Path**. Skip error cases unless they are the primary focus of the feature.
- **Skips:** Feature Decomposition Tree, Edge Case Brainstorming, RBT Matrix, Self-Criticism Loop.
- **Output suffix:** `testcases-hl`
- **Use when:** Hotfix validation, CI/CD gate, quick build verification.

### Mode: `detail-level` (DET)
- **Goal:** Generate 30–80+ exhaustive test cases for Full Regression testing.
- **Focus:** All 6 Phases, BVA, Decision Tables, RBAC (5 roles), Data Integrity, Edge Cases.
- **Runs:** ALL mandatory sub-steps (Feature Decomposition, RBT, Edge Case Brainstorming, Self-Criticism).
- **Output suffix:** `testcases-det`
- **Use when:** Pre-release QA, first-time feature testing, compliance/acceptance testing.

> **Mode Selection Rule:** If no mode is specified by the user or workflow, the Agent MUST ask:
> *"Bạn muốn sinh bộ test cases ở cấp độ nào? high-level (5–15 cases) hay detail-level (30–80+ cases)?"*
> Do NOT default to either mode silently.

This skill covers the following test types for **web applications**:
- Functional Testing
- Integration Testing
- UI Testing
- Functional/End-to-End (E2E) Testing
- Acceptance Testing

## Execution Philosophy

**Risk-First Priority:** Agent MUST identify and prioritize high-business-risk areas (CRITICAL/HIGH) BEFORE writing any test case. Areas involving money, security, authentication, and core data flows MUST be designed first and in the most detail.

**Anti-Hallucination Guardrails (STRICT):**
1. **No Guessing:** If the UC document does NOT specify a detail (e.g., Loading state duration, color code, exact error message text, timeout value), Agent MUST NOT fabricate it. Instead, mark it as `[TBD - Not specified in UC]` in the Expected Result or Pre-condition. NEVER invent data that is not in the source document.
2. **Negative First:** When assessing risk, Agent must ask itself: "Hệ thống này dễ chết nhất ở đâu?" (Where is this system most likely to fail?). Generate test cases for those failure points FIRST.
3. **Cross-reference Mandatory:** Every Test Case MUST be traceable back to a specific line/section in the UC document. If a Test Case cannot be traced, it must be flagged with `[INFERRED - No direct UC reference]`.
4. **Verbatim Actions & Labels:** You MUST extract the exact button names, field labels, and UI text from the Requirement document. NEVER substitute them with generic names (e.g., do NOT change "Thêm mới" to "Lưu" based on common UI patterns). The Requirement is the Single Source of Truth.

## HIGH-LEVEL MODE (`high-level`)

When mode is `high-level`, follow this **simplified 3-step workflow**:

### HL-Step 1: Input Analysis
- Identify the highest version of the required input files.
- Read the provided documents and comprehend the use case.
- Read Reference Files from `.agents/skills/tc-design/references/`.
- Do NOT generate `.xlsx` files or write Python scripts in this step.

### HL-Step 2: Focused Drafting
Write a draft file at: `testcases/[UC-ID]/[UC-ID]_testcases-hl_draft.md`

**SKIP the following (DET-only activities):**
- ❌ Feature Decomposition Tree (Sub-step A)
- ❌ Risk-Based Assessment / RBT Matrix (Sub-step B)
- ❌ Edge Case Brainstorming (Sub-step C)
- ❌ Phase 4 (Functional Integration)
- ❌ Phase 5 (UI-Level Non-Functional Testing)
- ❌ Phase 6 (GUI & Visual Compliance)
- ❌ Self-Criticism Loop

**INCLUDE only:**
1. **Happy Path per sub-function:** 1 case each for Create, Read, Update, Delete (or equivalent).
2. **Critical Error per sub-function:** 1 case each (e.g., required field empty, duplicate key, unauthorized access).
3. **1 RBAC Check:** Admin succeeds + Unauthorized role is blocked.
4. **Quick Traceability:** Map each TC to at least one AC. Flag Critical ACs not covered and note them as "Deferred to DET".

**Standards still apply:** Action Keywords, Atomic Steps, State-based ER, Verbatim Actions & Labels.

**Hard cap: Maximum 15 test cases per UC.**

### HL-Step 3: Output Generation (.xlsx)
- Populate `Testcase_template.xlsx` using the draft.
- Check for existing `testcases-hl` versions. Increment if needed.
- Save to: `testcases/[UC-ID]/[UC-ID]_[feature-name]_testcases-hl_[YYYYMMDD]_v[N].xlsx`
- Delete the draft `.md` file.
- Write summary with deferred ACs list.

## DETAIL-LEVEL MODE (`detail-level`)

When mode is `detail-level`, follow the **full 5-step mandatory workflow** below:

### DET-Step 1: Input Analysis (MANDATORY)

- Identify the highest version of the required input files (UC Readiness Report, Scenarios). Always select the highest version number available.
- Read the provided documents and comprehend the use case in preparation for test case design.
- Read Reference Files: Locate and read all example files inside `.agents/skills/tc-design/references/`. Analyze their structure, column mapping, and specific wording conventions.
- Do NOT generate `.xlsx` files or write Python scripts in this step.

### DET-Step 2: Detailed Drafting (MANDATORY)

Write a draft file at: `testcases/[UC-ID]/[UC-ID]_testcases-det_draft.md`

**Sub-step A: Feature Decomposition (Text Mindmap - MANDATORY):**
Trước khi viết bất kỳ Test Case nào, Agent PHẢI tạo một sơ đồ cây phân rã tính năng (Feature Mapping) ở đầu file draft. Mục tiêu: đảm bảo AI nhìn thấy toàn bộ "bức tranh" trước khi đi vào chi tiết từng lá.
Format:
```
[UC-ID] Feature Name
├── Sub-function 1 (e.g., View List)
│   ├── Empty State
│   ├── Populated State
│   └── Pagination
├── Sub-function 2 (e.g., Create Record)
│   ├── Happy Path
│   ├── Validation Rules
│   └── Error Handling
├── Sub-function 3 (e.g., Edit Record)
└── Sub-function 4 (e.g., Delete Record)
```

**Sub-step B: Risk-Based Assessment (RBT - MANDATORY):**
Sau khi phân rã tính năng, Agent PHẢI gán nhãn rủi ro cho từng sub-function dựa trên Risk Matrix (xem mục Priority & Risk Matrix bên dưới). Ghi kết quả vào draft:
```
| Sub-function | Risk Level | Rationale |
|---|---|---|
| Create Record | CRITICAL (P0) | Ảnh hưởng trực tiếp đến data integrity |
| View List | MEDIUM (P2) | Chỉ hiển thị, không thay đổi dữ liệu |
```
Agent PHẢI viết Test Cases cho vùng CRITICAL/HIGH trước, sau đó mới đến MEDIUM/LOW.

**Sub-step C: Edge Case Brainstorming (MANDATORY):**
Agent PHẢI liệt kê ra các "Edge Cases" (Trường hợp biên) tiềm năng của Use Case này. Ví dụ:
- Mất kết nối mạng khi đang lưu dữ liệu.
- Session hết hạn khi đang nhập liệu giữa chừng.
- Nhập dữ liệu vượt quá dung lượng cho phép của Database.
- Trình duyệt bị refresh/back khi form chưa lưu.
- **Data Conflict (Xung đột dữ liệu / Concurrency):** Thao tác đồng thời trên cùng một bản ghi. Ví dụ:
  - 2 user cùng mở màn hình Edit và cùng bấm Lưu (kiểm tra cơ chế lock hoặc cảnh báo ghi đè).
  - 1 user đang Edit, 1 user khác (hoặc tab khác) thực hiện Delete bản ghi đó (khi user Edit bấm Save thì hệ thống xử lý thế nào?).
Agent phải ghi lại danh sách Edge Cases này ở đầu file draft trước khi bắt đầu viết Test Cases.

Follow 6 distinct phases to design test cases:

#### Phase 1: Screen Initialization (Static States)

The agent must verify the landing state of the screen before any user interaction occurs.

- Empty State (No Data): Verify the visibility and specific default attributes of every object on the screen.
  - Explicitly describe placeholders, empty-state icons, or "No Data" messages.
- Populated State (With Data): Verify the default appearance of all items.
  - Describe the default state (Enabled/Disabled) of all components relative to the data present.

#### Phase 2: Item Interactions (Component States)

Verify the behavior of individual UI components without triggering core business logic.

- Navigation & Reset: Check 'X' icons, 'Cancel' buttons, and 'Close' controls to ensure they exit or reset the view as intended.
- Screen Initialization Triggers: Verify that clicking functional buttons correctly opens or initializes the relevant sub-functions or popups.
- Component Verification:
  - Dropdowns: Verify clickability and ensure the correct list of values is displayed.
  - Textboxes/Buttons: Verify states (Clickable, Disabled, Read-only) based on the design or business rules.
- Navigation Tools: Check pagination behavior (Active page, Next/Prev arrows).

#### Phase 3: Core Functional Testing (Logic Analysis)

Apply systematic testing techniques for every specific function available on the screen:

- Happy Path: Execute the successful flow using valid inputs.
- Validation:
  - Required Fields: Ensure the system blocks saving if mandatory fields are empty.
  - Format: Check Email, Phone, Date, or specific data formats.
  - Range: Check character length or numeric limits.
  - Boundary Value Analysis (BVA): Test at the limits (Min, Max, Min-1, Max+1).
- Exception/Error Handling: Trigger and verify system responses to invalid logic or external errors.

- **Logic Mapping (Phân tích nhánh logic - Decision Table Testing):**
  - Agent MUST xác định tất cả các điểm rẽ nhánh (IF-ELSE) trong Requirement.
  - Mỗi nhánh logic PHẢI có ít nhất 1 Test Case tương ứng.
  - Sử dụng Decision Table Testing: Liệt kê các tổ hợp điều kiện đầu vào và kết quả mong đợi cho từng tổ hợp.
  - Ví dụ: Nếu Requirement nói "Nếu User có quyền Edit VÀ bản ghi ở trạng thái Draft → cho phép sửa", Agent phải tạo Test Case cho TẤT CẢ tổ hợp: (có quyền + Draft), (có quyền + Active), (không quyền + Draft), (không quyền + Active).

- **Data Type Integrity (Toàn vẹn kiểu dữ liệu):**
  - Kiểm tra ký tự đặc biệt: SQL Injection payloads (`' OR 1=1 --`), XSS payloads (`<script>alert('xss')</script>`).
  - Kiểm tra khoảng trắng: Leading/Trailing spaces (Trim space), Multiple spaces giữa các từ.
  - Kiểm tra kiểu dữ liệu không hợp lệ: Nhập chữ vào ô số, nhập số vào ô email, nhập ngày không hợp lệ (31/02/2025).
  - Kiểm tra Unicode/Emoji: Nhập ký tự tiếng Việt có dấu, Emoji, ký tự CJK vào các trường text.

#### Phase 4: Functional Integration

Verify the synergy between different functions on the same screen.

- **Combination Search (Tìm kiếm kết hợp):** Agent PHẢI thiết kế kịch bản kết hợp nhiều điều kiện cùng lúc để kiểm tra tính đúng đắn của logic (AND/OR). Ví dụ: Nhập Search keyword + Chọn Filter trạng thái + Sorting theo cột.
- **Action Impact:** How a 'Search' action affects 'Pagination', or how 'Deleting' an item updates the list count in real-time.

#### Phase 5: UI-Level Non-Functional Testing

- Security: Check for sensitive data masking (passwords) and input sanitization (SQL Injection/XSS) in UI fields.
- U X/Loading: Verify the presence of loading indicators (spinners) during data fetching and button debounce to prevent double-clicks.

#### Phase 6: GUI & Visual Compliance (Design-to-Code)

This phase focuses exclusively on visual fidelity:

- Design Alignment: Compare every object against the design file (Figma/Mockup) regarding position, color (HEX codes), spacing, and font sizes.
- Responsive Design: Verify the UI rendering across various screen resolutions and aspect ratios.

**Test Case sorting rules:**
Rule 1: Sheet-level separation:
- GUI test cases → `GUI` sheet only.
- Functional test cases → `FUNCTION` sheet only.

Rule 2: Within each sheet — sort by Phase (1→6):
- Phase 1: Screen Initialization (Static States)
- Phase 2: Item Interactions (Component States)
- Phase 3: Core Functional Testing (Logic Analysis)
- Phase 4: Functional Integration
- Phase 5: UI-Level Non-Functional Testing
- Phase 6: GUI & Visual Compliance (Design-to-Code)

Rule 3: Within each Phase — sort by logical flow:
- Happy path first, then validation, then error/exception cases.

**Test Case Instruction rules:** Following 4 rules below:
Rule 1: Naming & Identifier Rules
TC ID: Always strictly adhere to the format: TC_[UC-ID]_[Type]_[Incremental Number].
Example: TC_UC001_GUI_01, TC_UC001_FUNC_02.

Test Title: Must begin with a verification verb (Check, Confirm, Verify).
Must include the scenario context. For example: ... in case no data, ... in case select [Value].

Rule 2: UI Notation Standard
The Agent must utilize specific notations to differentiate on-screen components:
[Square Brackets]: Use for interactive components such as Buttons, Menus, Tabs, Icons. (Example: [Create Crawl Job] button, [X] icon).
"Double Quotes": Use for Labels, Placeholders, input values, or selected values from a list. (Example: "Platform" dropdown, "Rule 3: Content Logic (STRICT MACHINE-READABLE STRUCTURE)
All fields (Pre-conditions, Test Steps, Expected Result) MUST adhere to the following rules:

**A. Sử dụng bộ "Động từ chuẩn hóa" (Action Keywords)**
Bắt đầu mỗi bước bằng một từ khóa trong ngoặc vuông để định nghĩa hành động:
- `[Navigate]`: Di chuyển đến URL/Menu.
- `[Input]`: Điền dữ liệu vào field (Kèm theo: [Input] "Field Label" = "Value").
- `[Click]`: Tác động vào Button, Icon, Link, Checkbox.
- `[Select]`: Chọn giá trị từ Dropdown/List.
- `[Verify]`: Kiểm tra trạng thái hiển thị hoặc giá trị (Assertion).
- `[Hover]`: Rê chuột lên đối tượng.
- `[Upload]`: Tải tệp tin lên hệ thống.

**B. Nguyên tắc "Vật thể + Vị trí" (Object + Location)**
- Luôn mô tả vị trí để tránh nhầm lẫn locator: `[Click] button "Edit" in the row containing text "HAN"`.
- Nhắc tên Label trong ngoặc kép: `[Input] "Airport Code" = "HAN"`.

**C. Dữ liệu "Tường minh" (Explicit Data)**
- Tuyệt đối không để AI tự suy luận dữ liệu. Ghi rõ giá trị cụ thể.
- Sử dụng Dynamic Data Placeholders `{{ }}` cho các giá trị động: `[Input] "Username" = "Auto_{{$timestamp}}"`.

**D. Cấu trúc "Bước Nguyên Tử" (Atomic Steps)**
- Mỗi dòng chỉ chứa DUY NHẤT một hành động.
- Ví dụ:
  1. [Input] "Username" = "admin"
  2. [Input] "Password" = "123"
  3. [Click] button "Login"

**E. Expected Result: Phải là "Trạng thái" (State-based)**
Mô tả kết quả dựa trên các tầng trạng thái:
- `[UI]`: Trạng thái hiển thị (Pop-up "Success" xuất hiện, Button bị disabled).
- `[API]`: Phản hồi từ server (Status 201, Response JSON chứa ID mới).
- `[DB]`: Trạng thái dữ liệu (Table "m_airports" xuất hiện bản ghi có code = "HAN").

**F. Cột "Context/Pre-condition" (4-Layer Structure)**
- Luôn xác định rõ bối cảnh trước khi chạy:
  1. [Environment] Browser/Network state.
  2. [Auth] Authenticated as [Role].
  3. [Data State] Existing record/Clean state.
  4. [System Config] Page current location (e.g., On "Airport Category" page).

Rule 4: State Transition Logic (STRICT)
Agent PHẢI mô tả trạng thái của đối tượng **trước** và **sau** khi thực hiện hành động.
- Pre-condition: "Existing record [ID: 123] with status [Draft]".
- Expected Result:
  1. [UI] Success Toast "Updated" appears.
  2. [UI] Record status changes to [Active].
  3. [DB] Field "status" in table "X" is updated to 1.
 active imperative verbs from the Cheat Sheet (Input, Click on, Select, Direct to, etc.).

Expected Result (UI Verification & Logic):
- Format: Numbered steps starting with verbs.
- Do not write generic statements like "System works correctly".
- Must explicitly describe the changed state of the UI/System using the Cheat Sheet verbs, covering up to 3 layers if applicable:
  - **Layer 1:** UI Verification (Màu sắc, vị trí, hiển thị).
  - **Layer 2:** Data Verification (Dữ liệu lưu vào DB đúng không, hiển thị đúng không).
  - **Layer 3:** Business Logic (Quy trình nghiệp vụ có bị gãy không).
- Example: "1. Hiển thị màn hình Dashboard; 2. Button Login chuyển sang Loading state trước khi chuyển trang."

Rule 4: State Transition Logic (STRICT)
Đối với các dự án quản lý thông tin, Agent PHẢI mô tả trạng thái của đối tượng (Object State) **trước** và **sau** khi thực hiện hành động.
- Pre-condition phải ghi rõ trạng thái ban đầu của đối tượng (e.g., "Existing record [ID: 123] with status [Draft]").
- Expected Result phải mô tả trạng thái mới của đối tượng SAU hành động, bao gồm:
  - Pop-up xác nhận hiển thị (nếu có).
  - Sự thay đổi trên UI (dữ liệu biến mất khỏi danh sách, trạng thái chuyển từ [Draft] sang [Active]).
  - Thông báo phản hồi (Display Success Toast / Trigger Validation Error).
  - Tổng số lượng bản ghi (Total count) thay đổi tương ứng (tăng/giảm).
- Ví dụ (Xóa bản ghi):
  - Before: Record [ID: 123] with status [Active] exists in list. Total count = 10.
  - After:
    1. Pop-up xác nhận "Bạn có chắc chắn muốn xóa?" appears.
    2. Click on [Confirm] → Record [ID: 123] disappears from list on UI.
    3. Display Success Toast "Xóa thành công".
    4. Total count updates to 9 via real-time update.

**Deep Understanding Highlights (Tư duy QA Chuyên sâu - MANDATORY):**
- **Negative over Positive:** Ưu tiên mô tả chi tiết các bước gây lỗi (Negative cases) để hệ thống không bị crash hoặc lộ thông tin nhạy cảm.
- **Context-Awareness:** Khi viết Test Case cho màn hình "Chỉnh sửa" (Edit), Pre-condition phải luôn bao gồm việc đã có dữ liệu sẵn và dữ liệu đó đang ở trạng thái cho phép sửa.
- **Cross-field Validation:** Nếu trường A phụ thuộc vào trường B (ví dụ: Chọn Thành phố rồi mới chọn Quận/Huyện), Agent PHẢI thiết kế Case để kiểm tra tính logic của sự phụ thuộc này.
- **UI Consistency:** Luôn kiểm tra tính nhất quán (ví dụ: Màu sắc nút [Save] ở trang thêm mới phải giống hệt trang chỉnh sửa).

**Priority & Risk Matrix (AI-RBT):**
Agent PHẢI sử dụng Ma trận Rủi ro dưới đây để gán Priority cho từng Test Case. Priority được xác định bởi công thức: **Impact × Likelihood**.

| Risk Level | Priority | Tiêu chí đánh giá (Impact × Likelihood) | Chiến lược viết TC |
| --- | --- | --- | --- |
| CRITICAL | P0 | Ảnh hưởng trực tiếp đến tiền bạc, bảo mật, luồng chính (Auth, Payment, Core CRUD). | Viết chi tiết từng bước (Step-by-step), bao gồm cả UI/Data/Logic (3 Layers). |
| HIGH | P1 | Tính năng quan trọng, tần suất sử dụng cực cao (Search, Filter, Main List). | Tập trung vào Boundary Value Analysis, Happy Path, và Decision Table. |
| MEDIUM | P2 | Tính năng hỗ trợ, ít rủi ro về dữ liệu (Sorting, Pagination, Export). | Viết kịch bản tổng quát, tập trung vào Validation rules. |
| LOW | P3 | Lỗi thẩm mỹ, UI nhỏ, ít người dùng (Tooltip, Color, Font). | Kiểm thử theo Checklist hoặc GUI static states. |

### DET-Step 3: Self-Review & Traceability Matrix

**Sub-step A: Self-Criticism Loop (MANDATORY before Traceability):**
Sau khi hoàn thành draft Test Cases, Agent PHẢI tự đánh giá chất lượng bộ TC vừa tạo bằng cách tự hỏi:
1. Có kịch bản Negative/Edge Case nào còn thiếu không? (Liệt kê tối thiểu 3 điểm yếu tiềm năng).
2. Có Test Case nào Expected Result quá chung chung, vi phạm quy tắc "No Guessing" không?
3. Có Acceptance Criteria nào từ UC chưa được bao phủ bởi bất kỳ TC nào không?

Agent PHẢI ghi kết quả Self-Review vào cuối file draft dưới dạng:
```
## Self-Review Results
| # | Weakness Found | Action Taken |
|---|---|---|
```
